### Typography and content

- **Do:** Use Inter or a strong native system sans stack.
- **Do:** Use 12–13px for metadata, 13–14px for controls and operational rows, and 15–16px for narrative copy.
- **Do:** Use semibold restraint for section and page titles.
- **Do:** Use tabular numerals for aligned dates, currency, metrics, counts, and durations.
- **Do:** Use concrete object names and task-based page titles.
- **Do:** Use verb-first button labels that name the outcome.
- **Do:** State errors with cause, impact, and next step.
- **Do:** Explain empty states with why, how data appears, and one next action.
- **Do:** Use consistent vocabulary and date/number formats.
- **Do:** Keep essential instructions in readable contrast.
- **Don't:** Use decorative display fonts in routine application surfaces.
- **Don't:** Use tiny low-contrast metadata to create an artificial sense of refinement.
- **Don't:** Use vague primary labels such as Continue when a precise action is available.
- **Don't:** Use marketing language inside operational controls.
- **Don't:** Hide essential instructions only inside tooltips.
- **Don't:** Use unexplained internal implementation labels as user-facing status.
- **Don't:** Write fabricated conversational filler for automation or agent output.
- **Don't:** Truncate status, identity, or critical deadlines before secondary titles.

### Controls and components

- **Do:** Use 28px compact, 32px normal, and 36px emphasis control tiers.
- **Do:** Use at most one dominant primary action in a local region.
- **Do:** Give icon-only controls accessible names and tooltips when not self-explanatory.
- **Do:** Use visible hover, press, selected, and keyboard-focus states.
- **Do:** Use 36px standard inputs and 32px compact filters where appropriate.
- **Do:** Keep labels visible; use placeholders as examples rather than labels.
- **Do:** Use chips for metadata and filters, tabs for navigation, and badges for counts/state.
- **Do:** Use cards only for coherent independent tasks or information sets.
- **Do:** Use integrated paper shells for toolbar/body/footer components.
- **Do:** Let dense rows bleed to container edges when that improves scanning.
- **Do:** Use proportional confirmation based on risk and reversibility.
- **Do:** Keep loading and progress attached to the originating region.
- **Don't:** Default desktop buttons and inputs to 40–48px.
- **Don't:** Use multiple accent-filled buttons in the same local region.
- **Don't:** Place a ring of equally emphasized icon controls around every header.
- **Don't:** Style every chip like a selected tab.
- **Don't:** Use cards as the default wrapper for every section.
- **Don't:** Give toolbar, body, and footer separate rounded rectangles and shadows.
- **Don't:** Use a modal for routine single-property edits.
- **Don't:** Replace clear loading text with an ambiguous spinner-only button.
- **Don't:** Use green decoration for every valid field.
- **Don't:** Scatter card actions across inconsistent corners and hover zones.

### Interaction and state

- **Do:** Make every meaningful action keyboard operable.
- **Do:** Expose actions through a global command palette and context-aware command graph.
- **Do:** Use fuzzy search across actions, destinations, records, views, and recent items.
- **Do:** Keep tabs local and preserve the surrounding object context.
- **Do:** Update the smallest region capable of expressing a state change.
- **Do:** Preserve scroll position, selection, open inspector, and active tab during local mutations.
- **Do:** Support inline editing, compact popovers, and nearby inspectors for common properties.
- **Do:** Use split views, docking, and saved layouts for workbench tasks.
- **Do:** Provide Undo or easy reversal for low-risk reversible actions.
- **Do:** Show impact previews and rollback paths for broad or high-risk changes.
- **Do:** Group and deduplicate notifications; escalate only material events.
- **Don't:** Force two or three navigation steps to edit a routine property.
- **Don't:** Redraw the entire page after posting a comment or changing a local filter.
- **Don't:** Use full-screen loading for a local mutation.
- **Don't:** Destroy layout state when entering or leaving focus mode.
- **Don't:** Hide basic orientation in the name of progressive disclosure.
- **Don't:** Use artificial wizard steps when the workflow is not truly sequential.
- **Don't:** Use a scrolling transcript as the only representation of structured work, evidence, or project state.
- **Don't:** Animate layout merely to make the interface feel active.
