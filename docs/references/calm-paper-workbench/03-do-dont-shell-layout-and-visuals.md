## Canonical do/don’t rules

### Shell and navigation

- **Do:** Keep the global shell stable while switching among overview, list, board, timeline, updates, editor, browser, and saved views.
- **Do:** Assign one owner to global navigation: sidebar, activity rail, or public header.
- **Do:** Use a 240px standard sidebar with 28px navigation rows for authenticated SaaS applications.
- **Do:** Use a slim activity rail plus contextual navigator for deep workbench products.
- **Do:** Allow sidebars and inspectors to collapse completely in focus-heavy workflows.
- **Do:** Use a location bar for current-object identity and stable object actions.
- **Do:** Use a separate view bar for representation, filters, grouping, display, and inspector controls.
- **Do:** Omit a bar when it has no semantic role or meaningful controls.
- **Do:** Keep navigation hierarchy shallow and preserve consistent label baselines.
- **Do:** Use indentation for nested items rather than placing every level in a new card.
- **Do:** Place counts, locks, and compact indicators at the trailing edge.
- **Do:** Keep account/help/support utilities visually secondary to active work.
- **Don't:** Duplicate the same global destinations in both a full header navigation and a full sidebar.
- **Don't:** Create a huge sidebar that tries to expose the entire information architecture at once.
- **Don't:** Use multi-level accordion navigation as the default way to find routine destinations.
- **Don't:** Turn every navigation group into a rounded floating card.
- **Don't:** Use accent-colored active navigation unless semantic meaning requires it.
- **Don't:** Leave empty location or view bars merely to preserve a template.
- **Don't:** Make local tabs feel like route changes into another application.
- **Don't:** Reset shell state, selection, or scroll when only the body representation changes.

### Layout and density

- **Do:** Classify each screen as centered action canvas, document plus rail, full operational canvas, feed/conversation, settings, or workbench.
- **Do:** Compress comparison, sorting, triage, and throughput surfaces.
- **Do:** Relax navigation, narrative pages, onboarding, authentication, settings, and empty states.
- **Do:** Target a 760px reading column for narrative content.
- **Do:** Use a 320px compact inspector or 384px rich rail.
- **Do:** Convert a rail to an overlay before the reading column falls below about 640px.
- **Do:** Use 320px board columns and preserve horizontal scrolling.
- **Do:** Use a 304px fixed semantic pane for timelines and let time scroll independently.
- **Do:** Use fixed metadata zones in dense rows and cards.
- **Do:** Allow empty board columns and operational regions to remain genuinely empty.
- **Do:** Use 48–64px whitespace to focus centered actions and meaningful empty states.
- **Do:** Use 24–32px between major document sections and 12–16px inside modules.
- **Don't:** Make every page equally sparse or equally dense.
- **Don't:** Stretch a single focused form or action across the full workspace.
- **Don't:** Force operational data into a narrow reading column.
- **Don't:** Shrink board columns below 280px simply to avoid horizontal scrolling.
- **Don't:** Crush a document to keep an inspector permanently docked.
- **Don't:** Use giant dashboard cards to fill available space.
- **Don't:** Create decorative filler content for empty columns or states.
- **Don't:** Turn every mobile table row into an arbitrary card without considering the task.

### Visual foundations

- **Do:** Use a warm cream canvas and warm-white paper surfaces in the canonical theme.
- **Do:** Keep a compatibility neutral theme for existing products and author dark mode separately.
- **Do:** Use near-black warm ink instead of pure black for routine text.
- **Do:** Use one restrained interaction accent and small semantic colors.
- **Do:** Pair status color with text, icon, or shape.
- **Do:** Use a 4px fine grid and 8px dominant spacing rhythm.
- **Do:** Use radii of 4, 6, 8, 10, and 12px according to component scale.
- **Do:** Use borders, tonal layers, alignment, and sticky positioning before shadow.
- **Do:** Use tactile inset shadows on buttons and compact controls.
- **Do:** Use restrained paper lift only for meaningful independent modules.
- **Do:** Reserve the strongest shadow for menus, popovers, dialogs, and previews.
- **Do:** Limit interactive card hover movement to 1px.
- **Don't:** Use pure white as the entire product background.
- **Don't:** Use cool blue-gray outlines inside the Warm Paper theme.
- **Don't:** Flood pages with the accent color.
- **Don't:** Use large semantic-tinted page backgrounds for routine status.
- **Don't:** Use frosted glass, glossy panels, heavy gradients, or neon glow as the organizing principle.
- **Don't:** Stack paper shadows through nested cards.
- **Don't:** Use radii above 16px except for deliberate pills.
- **Don't:** Make every persistent panel look independently floating.
