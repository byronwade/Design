# Golden references and fixtures

References demonstrate the contract under realistic pressure; they do not override accessibility, accepted decisions, or production behavior. Prefer rendered, testable examples over mood boards.

## Golden screens and flows

| ID | Target/profile | Shell and pattern | Route/story/design link | Required fixtures and states | Baseline owner/status |
| --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |

A useful minimum covers application home, collection, object detail, form/settings, dense operational canvas, one consequential flow, one constrained layout, and applicable marketing or native entry surfaces.

## Fixture matrix

For each golden reference include realistic normal content, long/localized content, empty, loading, partial, error, permission, offline/stale, reduced motion, high contrast or appearance modes, and relevant input modes.

## Anti-references

| Pattern to reject | Why it fails | Approved alternative |
| --- | --- | --- |
| generic equal-card dashboard | no decision hierarchy | queue, list, or dominant module with supporting regions |
| duplicated header and sidebar navigation | two owners for the same location | one global navigation owner |
| page-local primitive | creates drift | shared approved component or gap proposal |

## Baseline changes

A changed screenshot is not automatically an improvement. Record the design decision, affected profiles/states, accessibility and interaction review, and approver before accepting an intentional visual baseline update.
