# Quality system

A design is not complete because it looks polished. Release quality requires correct situation, structure, capability, craft, momentum, trust, identity, coherence, and implementation evidence.

## Read order

1. `REVIEW.md` — definition of done, severity, and scorecard.
2. `EVIDENCE.md` — what proves each class of claim.
3. `RULES.json` — stable lintable rule IDs for automation and review.

## Release policy

- no blocker is acceptable
- every applicable scorecard category normally scores at least 3 of 4
- automated checks supplement manual interaction and visual review
- a changed visual baseline requires explicit approval and evidence
- an exception identifies rule, reason, scope, owner, accessibility impact, and expiration or migration
- a clean surface receives a concise pass; reviewers do not invent findings to fill a report
