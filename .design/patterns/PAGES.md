# Page patterns

## Application home

Surface current work, attention, meaningful progress, recent objects, and a clear return path. Use a dashboard only when several independent signals support a recurring decision. Prefer a strong queue, list, or activity structure over a generic grid of equal cards.

## Collection or list page

Identity → view controls → collection → selection and bulk state → pagination or continuation. Search, active constraints, sorting, grouping, columns, display mode, empty state, refresh, and saved views preserve the current shell and URL-addressable state where appropriate.

## Object detail page

Object identity and object actions remain stable while the body exposes summary, sections, resources, activity, relationships, and optional inspector. Preserve return location and collection selection. Local tabs do not replace object identity.

## Create or edit page

Title and consequence → grouped fields → validation → review when risk requires it → stable commitment region. Preserve input through error and navigation. On constrained layouts, keep a single readable path rather than preserving a desktop two-column arrangement.

## Dashboard or analysis page

Every module answers a question or supports an action. State time range, source, freshness, partial failure, comparison basis, and drill-down. Avoid duplicated giant metrics, decorative charts, and equal-card layouts without a decision model.

## Activity or conversation page

Keep object or channel identity stable, provide chronological grouping and durable links, distinguish unread/new state, and place the composer at the point of insertion. Threads, mentions, evidence, and actions remain structured and inspectable.

## Marketing page

Audience/problem/value → proof → mechanism or workflow → capability → trust → conversion. Use specific claims, real product evidence, semantic content, responsive media, shallow navigation, and one dominant conversion per region. Do not import application density or authenticated shell chrome.

## Authentication, onboarding, and first run

Keep the task focused, explain why information or permission is needed, minimize navigation, preserve progress, and make exit/recovery clear. First run demonstrates value with one relevant action rather than decorative filler or an artificial wizard.

## Error, permission, and unavailable page

Explain what is unavailable, why, affected scope, retained state, next action, and support or diagnostic reference when needed. Do not make permission denial look like no data, or fatal failure look like a temporary empty state.
